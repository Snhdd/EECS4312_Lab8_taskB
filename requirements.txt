## 1. Problem Understanding - Describe the problem your system solves.

System purpose:
The system manages registration for a single event with a fixed capacity. It allows users to register for the event, places extra users on a waitlist when the event is full, supports cancellations, and promotes waitlisted users when spots become available.

Inputs to the system:
- Event capacity
- User registration requests
- User cancellation requests
- User status queries

Outputs produced by the system:
- Registration result for a user (registered or waitlisted with position)
- Updated waitlist positions
- Promotion of waitlisted users after cancellations
- User status information
- Error messages for duplicate requests or missing users

Primary goal of the system:
The primary goal of the system is to manage event registrations in a deterministic, reliable, and transparent way while respecting capacity limits, preventing duplicates, and preserving fair FIFO waitlist behavior.

---

## 2. Assumptions - List assumptions about the environment, users, or inputs.

A1: The system manages registration for only one event at a time.  
A2: Event capacity is provided as a non-negative integer.  
A3: Each user is uniquely identified by a non-empty string user ID.  
A4: A user may appear in only one state at a time: registered, waitlisted, or none.  
A5: Registration and cancellation requests are processed in the order they are received.  
A6: Waitlist ordering must follow FIFO behavior.  
A7: Users may query their status at any time.  
A8: Invalid operations such as duplicate registration or cancellation of a missing user must return explicit errors.

---

## 3. Functional Requirements - Describe the main behaviors the system must perform.

FR1: The system shall register a user directly if the event has available capacity.

FR2: The system shall place a user on the waitlist if the event is full.

FR3: The system shall reject duplicate registration requests for users who are already registered or waitlisted.

FR4: The system shall cancel a registered or waitlisted user when requested.

FR5: The system shall automatically promote the earliest waitlisted user when a registered user cancels and capacity becomes available.

FR6: The system shall return the current status of a user as registered, waitlisted with position, or none.

FR7: The system shall preserve deterministic FIFO ordering in the waitlist and promotions.

FR8: The system shall return explicit error feedback when an invalid cancellation or invalid registration request occurs.
